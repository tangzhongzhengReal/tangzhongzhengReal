Taro 下拉刷新

一个 BUG

小程序在自定义导航栏的时候，会把 page 的高度从手机屏幕最顶端开始算起。

大家做过的自定义导航栏的都知道，这个自定义导航栏其实是由两部分组成的，一部分是 fixed 定位的，使其一直显示到屏幕的最顶端假装是原生的导航栏。另一部分是做 padding 的，使 page 高度从导航栏下面开始算起。

搞笑的地方来了：
当调用小程序下拉刷新接口的时候，在 iOS 端，这个 padding 部分会被拉下来，fixed 的部分是不动的。但是 loading 动画会被 fixed 部分遮挡。机智如我，就想 padding 部分拉下来了，那交互动画我就放在 padding 部分吧。咔咔一写，开发工具完美，iOS 真机调试也很好。当我拿起 android 手机测试的时候，瞬间崩溃了。

在 android 手机上，下拉刷新会把 fixed 部分也拉下来，原生 loading 动画在屏幕最顶端显示！！！

奔溃了。

ScrollView

试着用 ScrollView 的 bindscrolltoupper 来实现，upper-threshold 属性可以设置负值。
这个方法 iOS 上可以用的，iOS 可以可以下拉到负值，但是 android 上是不可以的。

ScrollView thouchMove

这个其实是参考新浪微博小程序实现的，具体效果和他的一样。
thouchMove 这个函数发生在 touchStart 和 touchEnd 之间。在 ScrollView 组件上，当在组件内滑动的时候是 thouchMove 不会触发的。只有滑到边缘才会触发。

核心思路是：
1）在 scroll 上方设置一个 div，当刷新到时候显示，其他时候隐藏。
2）判断是否下拉
3）下拉之后显示 loading 部分，当异步取到数据之后，设置为刷新成功。
4）短暂等待后，隐藏 loading 部分。

判断是否下拉的逻辑

还有一点就是，当异步函数触发完成之后，要去隐藏掉下拉刷新的 bar。但是这个异步函数是从别的组件传入的，所以要在传入的方法里加一句回调。
